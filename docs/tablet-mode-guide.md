# BH4HAPtool 平板模式开发说明

## 1. 目标

平板模式用于在横屏大屏设备上提供更稳定的双栏体验，避免所有内容继续沿用手机纵向堆叠布局。

设计目标如下：

- 在横屏宽屏场景下自动切换为双栏布局
- 左侧突出主要互动区或游戏区
- 右侧集中放置控制栏、状态信息、说明和设置
- 手机竖屏与普通窄屏场景继续保留原有纵向布局

---

## 2. 触发条件

当前平板模式由共享适配层统一判断，代码位于：

- `core/toolkit/src/main/java/com/example/bh4haptool/core/toolkit/ui/ToolPaneMode.kt`

核心规则：

- 必须为横屏：`screenWidthDp > screenHeightDp`
- 长宽比达到阈值：`screenWidthDp / screenHeightDp >= 1.45f`

对应入口：

- `rememberToolPaneMode()`

返回结果中最重要的字段是：

- `isTabletMode`

---

## 3. 布局规则

平板模式统一采用左右双栏：

- 左栏：主互动区 / 主游戏画面
- 右栏：控制栏 + 信息栏 + 设置栏

比例约定：

- 左栏占 70%
- 右栏占 30%

共享布局容器：

- `TabletTwoPane(...)`

其职责：

- 应用 `Scaffold` 的 `innerPadding`
- 自动构建 70 / 30 双栏
- 右栏默认支持独立纵向滚动

---

## 4. 共享 API

### 4.1 `rememberToolPaneMode`

用途：

- 读取当前屏幕宽高
- 判断是否进入平板模式

示例：

```kotlin
val paneMode = rememberToolPaneMode()

if (paneMode.isTabletMode) {
    // 平板双栏
} else {
    // 手机纵向
}
```

### 4.2 `TabletTwoPane`

用途：

- 在平板模式下承载左右内容区

示例：

```kotlin
TabletTwoPane(contentPadding = innerPadding) {
    // 左侧主区域
} secondary@{
    // 右侧控制和信息区域
}
```

---

## 5. 子应用接入规范

每个子应用的 `Route` 页面建议按以下方式接入：

1. 在 `Route` 顶部调用 `rememberToolPaneMode()`
2. `if (paneMode.isTabletMode)` 走双栏布局
3. `else` 保留当前手机纵向布局
4. 主区域只放“核心交互”
5. 控制、状态、帮助、设置统一移到右栏
6. 弹窗、对话框、SnackBar 等浮层逻辑保持在双栏外层

建议结构：

```kotlin
Scaffold(...) { innerPadding ->
    if (paneMode.isTabletMode) {
        TabletTwoPane(contentPadding = innerPadding) {
            // 主内容
        } secondary@{
            // 控制 / 状态 / 设置
        }
    } else {
        // 原手机布局
    }
}
```

## 6. 各类子应用的推荐分栏方式

### 6.1 棋盘 / 画布类

适用：

- 扫雷
- 推箱子
- 俄罗斯方块
- 抓小猫

推荐放置：

- 左栏：棋盘、画布、主要游戏区域
- 右栏：方向键、操作按钮、状态卡片、说明、设置

### 6.2 抽签 / 输入工具类

适用：

- 普通抽签
- 摇一摇抽签

推荐放置：

- 左栏：输入区、结果展示区
- 右栏：执行按钮、统计信息、传感器状态、提示信息

### 6.3 多阶段页面类

适用：

- 飞盘分组

推荐放置：

- 设置页：左侧表单，右侧启动操作和说明
- 罗盘页：左侧主罗盘，右侧传感器状态和操作按钮
- 结果页：左侧结果列表，右侧重开与返回操作

## 7. 开发注意事项

### 7.1 不要把所有内容都塞进左栏

左栏应该只承载“用户当前最关注的核心区域”，否则平板模式只是把手机页面拉宽，无法体现结构优势。

### 7.2 右栏优先保证信息密度和可滚动

右栏适合放：

- 操作按钮
- 当前状态
- 参数设置
- 规则说明
- 统计信息

当内容较长时，优先让右栏独立滚动，不要压缩左侧主区域。

### 7.3 保留手机模式现有交互

平板模式是增强，不应破坏手机模式原有布局和操作顺序。

### 7.4 对话框不要塞进右栏内部

例如：

- 关卡选择
- 确认框
- 权限提示

这类内容应继续作为页面外层弹窗处理，避免受双栏滚动影响。

---

## 8. 测试清单

每次接入平板模式后，至少检查以下场景：

- 手机竖屏：仍然保持原纵向布局
- 手机横屏：确认是否应进入平板模式
- 平板横屏：确认进入双栏模式
- 旋转屏幕：模式切换后无崩溃、无状态丢失
- 右栏长内容：可以独立滚动
- 左栏主区域：不会被压缩到不可用
- 弹窗类功能：显示位置正常，不受双栏影响

---

## 9. 后续演进建议

后续如果平板适配继续扩展，建议统一沿着以下方向演进：

- 将更多通用布局逻辑继续下沉到 `core/toolkit/ui`
- 为不同类型页面抽象出更明确的双栏模板
- 根据实际设备体验再调整长宽比阈值
- 逐步补齐各子应用的平板模式截图和回归检查项

---

## 10. 结论

平板模式的核心不是“把页面放大”，而是“重新组织信息层级”。

在本项目中，统一遵循下面这条原则即可：

- 左边放主画面
- 右边放控制和信息
- 横屏且长宽比足够大时自动启用
